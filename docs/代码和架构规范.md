## 2. 分层职责

### 2.1 Routes（路由层）
- **职责**：按业务模块注册子路由，决定 URL 路径与 HTTP 方法。
- **允许**：导入 Controller、Middleware。
- **禁止**：导入 Service；包含业务逻辑。

### 2.2 Controllers（控制器层）
- **职责**：解析请求参数、校验输入、调用 Service、返回统一响应。
- **允许**：导入 Service、Middleware、Common/Response、Schema（Zod 校验）。
- **禁止**：直接执行 SQL；直接调用外部 HTTP API；包含业务规则。

### 2.3 Services（服务层）
- **职责**：封装业务逻辑与数据访问，对外暴露清晰的业务接口。使用 Drizzle ORM 操作数据库。
- **允许**：导入 Common（数据库、API 客户端、工具）；调用其他 Service。
- **禁止**：操作 `req` / `res`；导入 Controller 或 Route。

### 2.4 Middleware（中间件层）
- **职责**：在请求到达 Controller 前处理横切关注点（认证、日志、预处理）。
- **允许**：导入 Common 与 Service。
- **禁止**：导入 Controller 或 Route。

### 2.5 Common（公共层）
- **职责**：纯工具、配置、基础设施、通用响应、Drizzle ORM 数据库连接。
- **允许**：只导入标准库和第三方库。
- **禁止**：导入本项目任何上层目录（routes / controllers / services / middleware / tasks）。

### 2.6 Schemas（校验层）
- **职责**：使用 Zod 定义请求参数的校验规则，供 Controller 调用。
- **允许**：只导入标准库和 `zod`。
- **禁止**：导入本项目任意层级（保持纯校验逻辑）。

## 3. 依赖方向

上层可以依赖下层，下层禁止反向依赖上层：

```
routes → controllers → services → common
routes → controllers → schemas
middleware → services / common
tasks → services / common
```

**Common 层绝对不允许出现指向 controllers / services / routes / middleware / tasks 的 import。**

## 4. 命名规范

### 4.1 文件命名
- 使用 **kebab-case**（短横线连接的小写字母）。
- 示例：`require-login.ts`、`update-avatar.ts`、`server-error.ts`
- 特殊情况：模块目录内只有单一入口文件时，可用 `index.ts`。

### 4.2 函数与变量命名
- 使用 **camelCase**。
- **禁止**使用无意义的后缀，如 `API`、`Handler`、`Func`。
- 示例：`getAnimeByID`、`requireLogin`、`notFound`

### 4.3 类与接口命名
- 使用 **PascalCase**。
- 示例：`UserData`、`AnimeRow`

## 5. 响应规范

统一使用 `common/response/` 下的工具函数返回响应，保持格式一致：

```ts
res.status(200).send({ code: 200, message: "成功", data: ... });
res.status(400).send({ code: 400, message: "请求参数错误" });
```

| 函数 | 状态码 | 用途 |
|------|--------|------|
| `success(res, data?, message?)` | 200 | 正常响应 |
| `badRequest(res, message?)` | 400 | 参数缺失或格式错误 |
| `unauthorized(res, message?)` | 401 | 未登录 |
| `forbidden(res, message?)` | 403 | 无权限 |
| `notFound(res, message?)` | 404 | 资源不存在 |
| `serverError(res, message?)` | 500 | 服务器内部错误 |

## 6. 数据库访问

- **禁止**在 Controller 层直接执行 SQL。
- 数据库操作统一使用 Drizzle ORM，通过 `common/database/connection.ts` 导出的 `db` 实例，在 Service 层完成。
- 表定义文件位于 `common/database/schema/` 下，每张表一个文件。
- Drizzle 迁移文件位于 `drizzle/` 目录，由 `drizzle-kit` 管理。
- 启动时自动执行 `drizzle/` 下的迁移，无需手动运行 SQL 脚本。

## 7. 请求参数校验

- Controller 应在调用 Service 前使用 Zod schema 校验输入参数。
- Schema 文件统一放在 `schemas/v2/` 下，按业务模块分目录。
- Controller 通过 `common/tools/validate.ts` 中的校验中间件调用 schema。

## 8. 外部 API 调用

- 禁止在 Controller 层直接调用外部 HTTP API。
- 外部 API 客户端统一封装在 `common/api-clients/` 下，Service 层按需使用。
- `common/api-clients/` 本身不得依赖业务代码（如从 controllers 读取配置）。

## 9. 类型定义

- 数据库表对应的类型定义统一放在 `types/models.d.ts`。
- Express 扩展类型（如 `req.user`）放在 `types/express.d.ts`。
- Service 层函数参数与返回值应尽可能使用 TypeScript 类型。

## 10. 废弃代码处理

- 确认未使用的模块直接删除，不保留空目录。
- 重构完成后清理旧文件与旧目录，避免历史包袱。

## 11. 验证 checklist

新增或修改代码时，请确认：

- [ ] 文件命名使用 kebab-case，函数使用 camelCase。
- [ ] Controller 不直接调用 SQL 或外部 API。
- [ ] `common/` 内不存在指向 `controllers` / `services` / `routes` / `middleware` / `tasks` 的 import。
- [ ] Service 不操作 `req` / `res`。
- [ ] 路由层只导入 Controller 和 Middleware。
- [ ] 所有 HTTP 响应使用 `common/response/` 的统一函数。
- [ ] 请求参数使用 `schemas/v2/` 下的 Zod schema 进行校验。
